ReasonDB is built as a Rust workspace with four main crates: `reasondb-core`, `reasondb-ingest`, `reasondb-plugin`, and `reasondb-server`. This page explains the system architecture and how components interact.

## reasondb-core

The core library provides the fundamental abstractions:

### Data Models

```rust
// Document: Top-level container
struct Document {
    id: String,
    title: String,
    root_node_id: String,
    table_id: String,
    total_nodes: usize,
    max_depth: u8,
    tags: Vec<String>,
    metadata: HashMap<String, Value>,  // All custom fields (author, etc.)
}

// Node: Tree node with content and summary
struct Node {
    id: String,
    document_id: String,
    parent_id: Option<String>,
    children_ids: Vec<String>,
    title: String,
    content: Option<String>,
    summary: String,
    depth: u8,
}

// Table: Document collection
struct Table {
    id: String,
    name: String,
    description: Option<String>,
    metadata: HashMap<String, Value>,
    document_count: usize,
    total_nodes: usize,
}
```

The storage layer uses `redb` for ACID-compliant persistence:

### LLM Interface

The `ReasoningEngine` trait abstracts LLM providers:

```rust
#[async_trait]
pub trait ReasoningEngine {
    async fn evaluate_relevance(
        &self,
        query: &str,
        summaries: &[NodeSummary],
    ) -> Result<Vec<RelevanceScore>>;

    async fn extract_answer(
        &self,
        query: &str,
        content: &str,
    ) -> Result<ExtractedAnswer>;

    async fn generate_summary(
        &self,
        content: &str,
    ) -> Result<String>;
}
```

### Search Engine

The beam search algorithm for tree traversal:

<Mermaid>
{`stateDiagram-v2
    [*] --> GetRoots: Query received
    GetRoots --> EvaluateNodes: Get root summaries
    EvaluateNodes --> PruneLow: LLM scores relevance
    PruneLow --> SelectTop: Remove low confidence
    SelectTop --> CheckDepth: Keep top beam_width
    CheckDepth --> GetChildren: depth not at max
    CheckDepth --> ExtractAnswer: depth at max
    GetChildren --> EvaluateNodes: Get child summaries
    ExtractAnswer --> [*]: Return results`}
</Mermaid>

### RQL Parser

SQL-like query parsing with Pest grammar:

```
query = { select ~ from ~ where? ~ search? ~ reason? ~ group_by? ~ order_by? ~ limit? }

select = { "SELECT" ~ (star | field_list) }
from = { "FROM" ~ table_name }
where = { "WHERE" ~ condition }
search = { "SEARCH" ~ string_literal }
reason = { "REASON" ~ string_literal }
```

## reasondb-plugin

The plugin crate manages external process-based plugins for extraction, post-processing, chunking, and summarization:

```rust
struct PluginManager {
    registry: PluginRegistry,   // Discovered plugins by kind
    plugins_dir: PathBuf,       // Plugin discovery directory
}

struct PluginManifest {
    name: String,
    runner: RunnerConfig,       // command, args, timeout
    capabilities: Capabilities, // kind, formats, priority
}
```

Plugins are external executables (Python, Node.js, Bash, or compiled binaries) that communicate via JSON over stdin/stdout. See the [Plugin Guide](/guides/plugins) for details.

## Data Flow

### Ingestion Flow

<Mermaid>
{`sequenceDiagram
    participant Client
    participant Server
    participant Pipeline
    participant LLM
    participant Store

    Client->>Server: POST /v1/ingest/text
    Server->>Pipeline: Start ingestion
    Pipeline->>Pipeline: Extract & chunk text
    Pipeline->>Pipeline: Build tree structure
    loop For each node (bottom-up)
        Pipeline->>LLM: Generate summary
        LLM-->>Pipeline: Summary text
    end
    Pipeline->>Store: Save document & nodes
    Pipeline->>Store: Index for BM25
    Store-->>Server: Document created
    Server-->>Client: IngestResponse`}
</Mermaid>

### Search Flow

<Mermaid>
{`sequenceDiagram
    participant Client
    participant Server
    participant Engine
    participant LLM
    participant Store

    Client->>Server: POST /v1/search
    Server->>Engine: Execute search
    Engine->>Store: Get root node summaries
    loop Beam search until max_depth
        Engine->>LLM: Evaluate relevance
        LLM-->>Engine: Confidence scores
        Engine->>Engine: Prune low confidence
        Engine->>Engine: Select top beam_width
        Engine->>Store: Get child summaries
    end
    Engine->>LLM: Extract answers
    LLM-->>Engine: Extracted answers
    Engine-->>Server: SearchResponse
    Server-->>Client: Results + stats`}
</Mermaid>

## Technology Choices

| Component | Technology | Rationale |
|-----------|------------|-----------|
| Storage | `redb` | Pure Rust, ACID, fast writes |
| Text Index | `tantivy` | Rust Lucene, BM25 search |
| Extraction | Plugin system | Process-based; Python, Node.js, Bash, binaries |
| Serialization | `bincode` | Fast binary, zero-copy |
| Async | `tokio` | Industry standard, parallel I/O |
| HTTP | `axum` | Type-safe, ergonomic, fast |
| LLM | `rig-core` | Multi-provider abstraction |
| Docker | Alpine 3.21 | Minimal image with plugin runtimes |

## Performance Considerations

### Caching

- **Query Cache**: REASON queries are cached by query + table
- **LLM Responses**: Summaries stored with nodes (no re-generation)
- **Index**: BM25 index for fast keyword search

### Parallelism

- **Beam Search**: Multiple branches explored concurrently
- **Ingestion**: Async I/O for file processing
- **Tokio Runtime**: Multi-threaded async execution

### Storage Efficiency

- **Binary Encoding**: `bincode` for compact storage
- **Incremental Updates**: Only modified nodes written
- **ACID Transactions**: Consistent state on crashes

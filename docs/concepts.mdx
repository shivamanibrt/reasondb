ReasonDB isn't an incremental improvement—it's a fundamental rethinking of how databases should work in the AI era. This page explains the core concepts that make intelligent document retrieval possible.

## The Key Insight

**Documents have structure. That structure has meaning. Losing it is losing intelligence.**

When you read a legal contract, you don't scan every word hoping to stumble upon the termination clause. You look at the table of contents, navigate to the relevant section, and drill down. ReasonDB gives AI the same capability.

<Info>
**Hierarchical Reasoning Retrieval (HRR)** is ReasonDB's core innovation: preserving document structure and using LLMs to navigate it intelligently, rather than embedding everything into a flat vector space.
</Info>

## Documents Become Trees

When you ingest a document, ReasonDB transforms it into a navigable tree:

<Mermaid>
{`graph TD

    Root --> C["3. Payment Terms"]
    Root --> D["4. Termination"]

    C --> C2["3.2 Payment Schedule"]
    C --> C3["3.3 Late Penalties"]

**Why trees?**
- They mirror how documents are actually organized
- Each level provides increasing specificity
- Navigation is intuitive for both humans and AI
- Context is preserved at every level

## Nodes: The Building Blocks

Every piece of content in ReasonDB lives in a **node**. Nodes are the atomic units of knowledge.

| Property | Purpose | Example |
|----------|---------|---------|
| `id` | Unique identifier | `node_7f3a9b2c` |
| `title` | Section heading | "3.2 Payment Schedule" |
| `content` | Actual text | "Payments are due within 30 days..." |
| `summary` | LLM-generated navigation aid | "Specifies payment timing, grace periods, and invoicing requirements" |
| `depth` | Level in tree (0 = root) | `2` |
| `children` | Child node IDs | `["node_abc", "node_def"]` |
| `parent` | Parent node ID | `"node_xyz"` |

### Leaf vs Branch Nodes

<CardGroup cols={2}>
  <Card title="Branch Nodes" icon="folder">
    Have children. Contain section overviews and summaries. Used for navigation decisions.
  </Card>
  <Card title="Leaf Nodes" icon="file">
    No children. Contain the actual detailed content. Where answers are extracted.
  </Card>
</CardGroup>

**The summary is crucial.** When the LLM navigates the tree, it reads summaries to decide which branches are worth exploring. A good summary captures what questions this section can answer.

## LLM-Guided Traversal

This is where ReasonDB diverges from every other approach. Instead of retrieving then reasoning, ReasonDB **reasons to retrieve**.

### The Traversal Process

<Mermaid>
{`sequenceDiagram
    participant User
    participant ReasonDB
    participant LLM
    participant Tree
    
    User->>ReasonDB: "What are the late payment penalties?"
    
    ReasonDB->>Tree: Get root children summaries
    ReasonDB->>LLM: Which sections might answer this?
    LLM-->>ReasonDB: "Payment Terms" (0.92), "Termination" (0.31)
    
    Note over ReasonDB: Prune low-confidence branches
    
    ReasonDB->>Tree: Get "Payment Terms" children
    ReasonDB->>LLM: Which subsection is most relevant?
    LLM-->>ReasonDB: "Late Penalties" (0.97)
    
    ReasonDB->>Tree: Get leaf content
    ReasonDB->>LLM: Extract answer from content
    LLM-->>ReasonDB: "Late payments incur 1.5% monthly interest..."
    
    ReasonDB-->>User: Answer + confidence + path`}
</Mermaid>

### Why This Works Better

| Vector Search | ReasonDB Traversal |
|---------------|-------------------|
| Embeds query and chunks into same space | LLM reasons about relevance in natural language |
| "Similar" vectors might be irrelevant | LLM understands semantic intent |
| Loses document structure | Preserves and leverages structure |
| Same approach for all queries | Adapts depth based on query complexity |
| Black box retrieval | Explainable navigation path |

## Beam Search: Parallel Exploration

Real questions sometimes have answers in multiple places, or the best path isn't obvious upfront. ReasonDB uses **beam search** to explore multiple promising branches simultaneously.

```
Query: "What happens if I miss a payment?"

Beam Width: 3

Level 1 - Evaluate all sections:
  → "Payment Terms" (0.88) ✓ Keep
  → "Termination" (0.72) ✓ Keep  
  → "Definitions" (0.15) ✗ Prune
  → "Services" (0.08) ✗ Prune

Level 2 - Explore top 3 branches in parallel:
  → Payment Terms → Late Penalties (0.94) ✓
  → Payment Terms → Payment Schedule (0.45) ✓
  → Termination → Termination for Cause (0.68) ✓

Level 3 - Continue to leaves, extract answers
```

### Tuning Beam Search

| Parameter | Default | Effect |
|-----------|---------|--------|
| `beam_width` | 3 | More branches = more thorough, slower |
| `max_depth` | 10 | Deeper traversal for complex documents |
| `min_confidence` | 0.3 | Prune branches below this threshold |

```sql
-- A broad REASON query to explore many branches
SELECT * FROM contracts
REASON 'What are all the ways the contract can end?'
LIMIT 10
```

<Tip>
Increase `beam_width` (server config) for exploratory questions where answers might be scattered. Decrease for specific factual queries.
</Tip>

## The REASON Query Pipeline

The sections above explain how tree traversal works **within a single document**. But a real query spans an entire table with potentially millions of documents. When you run a `REASON` query, ReasonDB uses a **3-phase agentic pipeline** to progressively narrow down and deeply reason over documents.

<Mermaid>
{`flowchart LR
    Q["🔍 REASON Query"] --> P1
    
    subgraph P1["Phase 1: Candidate Selection"]
        direction TB
        BM25["BM25 Keyword Search"]
        BM25 --> C100["~100 candidates"]
    end
    
    P1 --> P2
    
    subgraph P2["Phase 2: LLM Ranking"]
        direction TB
        SUM["Read document summaries"]
        SUM --> RANK["LLM ranks by relevance"]
        RANK --> C10["Top ~10 documents"]
    end
    
    P2 --> P3
    
    subgraph P3["Phase 3: Deep Reasoning"]
        direction TB
        TREE["LLM tree traversal"]
        TREE --> ANS["Extract answers"]
    end
    
    P3 --> R["📄 Results + Answers"]
    
    style P1 fill:#1e1e2e,stroke:#89b4fa,color:#cdd6f4
    style P2 fill:#1e1e2e,stroke:#a6e3a1,color:#cdd6f4
    style P3 fill:#1e1e2e,stroke:#f38ba8,color:#cdd6f4`}
</Mermaid>

Each phase acts as a funnel — BM25 gives breadth, LLM ranking gives precision, and tree traversal gives depth.

### Phase 1: Candidate Selection

The first step is **fast, coarse retrieval** to narrow millions of documents down to ~100 candidates without any LLM calls.

```
Millions of documents
    ↓ BM25 keyword search
~100 candidate documents
```

ReasonDB picks a strategy in priority order:

1. If your query has a `SEARCH` clause, use its keywords for BM25 search
2. Otherwise, use the `REASON` query text itself as BM25 keywords
3. If neither produces results, fall back to metadata/tag filters (capped at 1,000 documents for safety)

This is purely lexical — no LLM calls, sub-millisecond latency.

### Phase 2: Agentic Summary Ranking

The ~100 candidates are too many for expensive deep reasoning. So ReasonDB asks the LLM to **rank documents by their summaries** — the title, root node summary, and tags of each document.

```
~100 candidates with summaries
    ↓ LLM ranks by relevance to query
Top ~10 most relevant documents
```

The LLM sees lightweight metadata (not full content) and returns a relevance score (0.0–1.0) plus reasoning for each document. Only the top N survive, where N is your `LIMIT` clause (default 10).

<Info>
If there are fewer candidates than the target limit, this phase is skipped entirely — no unnecessary LLM calls.
</Info>

### Phase 3: Deep Reasoning (Parallel)

This is the core agentic step. For each surviving document, the SearchEngine performs the **LLM-guided tree traversal** described in the earlier sections — beam search through the document hierarchy, with answer extraction at the leaves.

The key detail: documents are processed in **parallel batches** of up to 5 at a time, so deep reasoning happens concurrently across multiple documents.

```
Top 10 documents
    ↓ parallel tree traversal (5 at a time)
    ↓ LLM navigates each document tree
    ↓ extracts answers at leaf nodes
Ranked results with answers + confidence scores
```

For each document, the LLM:
1. Reads root children summaries → picks the most promising branches
2. Traverses deeper, pruning low-confidence paths (beam search)
3. At leaf nodes, verifies relevance and **extracts a direct answer** to the query
4. Returns the answer, confidence score, and the navigation path taken

### Result Assembly

After Phase 3 completes:
1. All matches are **sorted by confidence** (highest first)
2. Filtered by `min_confidence` if you specified `WITH CONFIDENCE > X`
3. Paginated via `LIMIT` / `OFFSET`

ReasonDB also applies **early termination** — if enough high-confidence results are found in early batches, remaining batches are skipped.

### Putting It All Together

Here's the full flow for a concrete query:

```sql
SELECT * FROM vendor_contracts
WHERE metadata.status = 'active'
SEARCH 'payment penalties'
REASON 'What happens if we miss a payment deadline?'
WITH CONFIDENCE > 0.5
LIMIT 5
```

| Phase | What Happens | Documents | LLM Calls |
|-------|-------------|-----------|-----------|
| **1. Candidate Selection** | BM25 searches "payment penalties" across active vendor contracts | 95,000 → 100 | 0 |
| **2. Summary Ranking** | LLM reads 100 document summaries, ranks by relevance to the question | 100 → 5 | 1 |
| **3. Deep Reasoning** | LLM traverses each document tree, extracts answers from relevant sections | 5 documents, ~20 nodes each | ~15 |
| **Result** | Sorted by confidence, filtered above 0.5 | 3-5 final answers | **~16 total** |

<Tip>
REASON query results are **cached** by the combination of reason query text and table ID. Repeated identical queries return instantly without any LLM calls.
</Tip>

## Tables: Organizing Knowledge

**Tables** are collections of documents. Think of them as namespaces or folders for your knowledge base.

```sql
-- Query across different collections
SELECT * FROM legal_contracts SEARCH 'termination' LIMIT 5
SELECT * FROM technical_docs REASON 'How do we handle authentication?'
```

**Why tables matter:**
- Scope searches to relevant collections
- Different retention and access policies
- Track document counts and usage
- Organize by department, project, or topic

## Querying with RQL

ReasonDB uses **RQL** (ReasonDB Query Language) — a SQL-like syntax with `SEARCH` and `REASON` clauses built in.

### SEARCH — Fast Keyword Lookup

BM25 full-text search for known terms:

```sql
SELECT * FROM legal_contracts
WHERE tags CONTAINS 'nda'
SEARCH 'confidentiality obligations'
LIMIT 10
```

**Best for:** Keyword matching, filtering by metadata, fast results (~50ms)

### REASON — LLM-Guided Retrieval

The [3-phase agentic pipeline](#the-reason-query-pipeline) kicks in: BM25 narrows candidates, the LLM ranks by summary, then tree traversal extracts precise answers.

```sql
SELECT * FROM legal_contracts
WHERE metadata.author = 'Legal Team'
REASON 'What happens to intellectual property if the contract is terminated?'
```

**Best for:** Complex questions, answer extraction, reasoning across sections

### Combining Everything

RQL lets you filter, search, and reason in a single query:

```sql
SELECT title, metadata.author FROM legal_contracts
WHERE tags CONTAINS ANY ('nda', 'msa')
  AND metadata.value_usd > 50000
SEARCH 'termination clause'
REASON 'What are the exit conditions?'
LIMIT 5
```

<Tip>
Use `SEARCH` when you know the keywords. Use `REASON` when you have a question. Combine both for maximum precision.
</Tip>

## The Ingestion Pipeline

Understanding how documents become trees helps you optimize for your use case:

<Mermaid>
{`flowchart TB
    A[📄 PDF / Word / HTML / Text / URLs] --> B[Extractor Plugin]
    B --> C[Post-Processor Plugin]
    C --> D[Chunk by Structure]
    D --> E[Build Hierarchical Tree]
    E --> F[LLM Summarizes Bottom-Up]
    F --> G[(Knowledge Base)]`}
</Mermaid>

1. **Extract**: Extractor plugins convert files and URLs to Markdown (built-in: [MarkItDown](https://github.com/microsoft/markitdown))
2. **Post-process**: Optional plugins clean up or transform the Markdown
3. **Chunk**: Split by headings, paragraphs, and semantic boundaries
4. **Build Tree**: Create hierarchy from document outline
5. **Summarize**: LLM generates summaries bottom-up (leaves first, then parents)
6. **Index**: Store with BM25 text search and tree structure

<Warning>
**Summary quality is critical.** Poor summaries lead to poor navigation. ReasonDB uses your configured LLM to generate summaries—better LLM = better summaries = better search.
</Warning>

## Scaling to Millions of Documents

The tree structure isn't just about understanding—it's the key to scaling. Here's why ReasonDB stays fast even with millions of documents.

### The Math: Why Trees Beat Flat Search

Consider a knowledge base with **1 million documents**, each containing **50 nodes** on average. That's **50 million nodes** total.

| Approach | What it searches | Nodes touched | Latency |
|----------|-----------------|---------------|---------|
| **Vector DB (top-k)** | All 50M embeddings | 50,000,000 | Slow, needs ANN index |
| **BM25 keyword** | All 50M text chunks | 50,000,000 | Fast but imprecise |
| **ReasonDB (tree)** | Summaries at each level | ~20–50 nodes | Fast AND precise |

**How?** The tree structure acts as a natural index. At each level, the LLM prunes irrelevant branches:

```
Level 0: 1,000,000 documents → LLM reads table-level summaries
         → Selects 3 most relevant documents

Level 1: 3 root nodes → reads section summaries
         → 9 children selected (3 per doc × beam_width=3)

Level 2: 9 nodes → reads subsection summaries
         → 12 candidates

Level 3: 12 leaf nodes → extract answers
         → 2–3 final results

Total: ~25 nodes visited out of 50,000,000
```

Each level is an **exponential filter**. The LLM never reads content it doesn't need to.

Imagine a law firm with 500,000 contracts ingested into ReasonDB:

**Query:** *"What late payment penalties does the Acme vendor agreement specify?"*

The LLM navigates:
1. **Tables** → "Vendor Agreements" is relevant (skips 499,905 irrelevant contracts)
2. **Documents** → "Acme Corp" matches (skips 94,999 others)
3. **Sections** → "§3 Payment" is relevant (skips Definitions, Services, Termination)
4. **Subsections** → "3.3 Late Penalties" is the answer (skips Fees, Schedule)

**Result**: 4 LLM calls. 8 nodes visited. Sub-second response. Out of 25 million total nodes.

### Why Not Just Use BM25 or Vectors?

BM25 for the same query would return every contract mentioning "late" or "penalty" — thousands of irrelevant results from employment contracts, NDAs, and leases. Vector search would find "similar" chunks but lose the structural context of *which contract* and *which section*.

ReasonDB combines the precision of structured navigation with the intelligence of LLM reasoning.

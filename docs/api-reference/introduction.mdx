The ReasonDB API gives you programmatic access to intelligent document storage, LLM-guided search, and structured queries. Build applications that understand documents, not just store them.

<Info>
**Interactive Playground**: Explore and test all endpoints at `http://localhost:4444/swagger-ui/`
</Info>

## Base URL

```
http://localhost:4444/v1
```

<Note>
Configure with `REASONDB_HOST` and `REASONDB_PORT`. Default: `localhost:4444`
</Note>

## Quick Reference

### Core Endpoints

| Endpoint | Purpose |
|----------|---------|
| `POST /v1/search` | **LLM-guided search** — Ask questions, get answers with reasoning |
| `POST /v1/query` | **RQL queries** — SQL-like queries with SEARCH and REASON clauses |
| `POST /v1/ingest/text` | **Ingest documents** — Transform text into navigable knowledge trees |
| `POST /v1/tables` | **Create collections** — Organize documents into searchable groups |

## Authentication

When auth is enabled, include your API key:

<Tabs>
  <Tab title="Authorization Header (Recommended)">
    ```bash
    curl -X POST http://localhost:4444/v1/search \
      -H "Authorization: Bearer rdb_live_xxxxx" \
      -H "Content-Type: application/json" \
      -d '{"query": "What are the payment terms?"}'
    ```
  </Tab>
  <Tab title="X-API-Key Header">
    ```bash
    curl -X POST http://localhost:4444/v1/search \
      -H "X-API-Key: rdb_live_xxxxx" \
      -H "Content-Type: application/json" \
      -d '{"query": "What are the payment terms?"}'
    ```
  </Tab>
</Tabs>

<Tip>
Authentication is optional in development. Enable for production with `--auth-enabled`. See [Authentication Guide](/guides/authentication).
</Tip>

## Complete API Reference

### Tables — Organize Your Knowledge

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/v1/tables` | Create a new table |
| `GET` | `/v1/tables` | List all tables |
| `GET` | `/v1/tables/{id}` | Get table details |
| `PATCH` | `/v1/tables/{id}` | Update table metadata |
| `DELETE` | `/v1/tables/{id}` | Delete table and all documents |
| `GET` | `/v1/tables/{id}/documents` | List documents in table |

### Ingestion — Build Knowledge Trees

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/v1/ingest/text` | Ingest text or Markdown |
| `POST` | `/v1/ingest/file` | Upload and process file (PDF, HTML, etc.) |
| `POST` | `/v1/ingest/url` | Fetch and ingest from URL |

### Search & Query — Find, Update, Delete

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/v1/search` | LLM-guided tree traversal with answer extraction |
| `POST` | `/v1/query` | Execute RQL statements (SELECT, UPDATE, DELETE) |

### Documents — Manage Content

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/v1/documents` | List all documents |
| `GET` | `/v1/documents/{id}` | Get document metadata |
| `DELETE` | `/v1/documents/{id}` | Delete document and nodes |
| `GET` | `/v1/documents/{id}/nodes` | Get all nodes in document |
| `GET` | `/v1/documents/{id}/tree` | Get full tree structure with summaries |

### Authentication — Secure Access

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/v1/auth/keys` | Create API key |
| `GET` | `/v1/auth/keys` | List all keys |
| `GET` | `/v1/auth/keys/{id}` | Get key details |
| `DELETE` | `/v1/auth/keys/{id}` | Revoke key |
| `POST` | `/v1/auth/keys/{id}/rotate` | Rotate key (new secret) |

## Common Workflows

### Workflow 1: Ingest and Search

```bash
# Step 1: Create a table
TABLE_ID=$(curl -s -X POST http://localhost:4444/v1/tables \
  -H "Content-Type: application/json" \
  -d '{"name": "Contracts"}' | jq -r '.id')

# Step 2: Ingest a document
DOC_ID=$(curl -s -X POST http://localhost:4444/v1/ingest/text \
  -H "Content-Type: application/json" \
  -d "{
    \"title\": \"Service Agreement\",
    \"table_id\": \"$TABLE_ID\",
    \"content\": \"# Service Agreement\\n\\n## Payment Terms\\n\\nPayment is due within 30 days...\"
  }" | jq -r '.document_id')

# Step 3: Ask questions
curl -X POST http://localhost:4444/v1/search \
  -H "Content-Type: application/json" \
  -d '{"query": "When is payment due?"}'
```

### Workflow 2: RQL Query with Reasoning

```bash
# Combine structured filters with LLM reasoning
curl -X POST http://localhost:4444/v1/query \
  -H "Content-Type: application/json" \
  -d '{
    "query": "SELECT * FROM Contracts WHERE metadata.status = '\''active'\'' REASON '\''What are the termination conditions?'\'' LIMIT 5"
  }'
```

### Workflow 3: Update and Delete via RQL

```bash
# Update documents matching a condition
curl -X POST http://localhost:4444/v1/query \
  -H "Content-Type: application/json" \
  -d '{
    "query": "UPDATE Contracts SET metadata.status = '\''archived'\'' WHERE metadata.status = '\''expired'\''"
  }'

# Delete documents matching a condition
curl -X POST http://localhost:4444/v1/query \
  -H "Content-Type: application/json" \
  -d '{
    "query": "DELETE FROM Contracts WHERE metadata.status = '\''abandoned'\''"
  }'
```

## Response Format

### Search Response (`/v1/search`)

```json
{
  "results": [{
    "node_id": "node_abc123",
    "document_id": "doc_xyz789",
    "path": [{"node_id": "node_root", "title": "Service Agreement", "reasoning": "Root"}],
    "content": "Payment is due within 30 days of invoice...",
    "answer": "Payment is due within 30 days of receiving the invoice.",
    "confidence": 0.94
  }],
  "stats": {
    "nodes_visited": 4,
    "nodes_pruned": 8,
    "llm_calls": 3,
    "total_time_ms": 1250
  }
}
```

### Query Response (`/v1/query`)

```json
{
  "documents": [{
    "id": "doc_xyz789",
    "title": "Service Agreement",
    "table_id": "tbl_legal",
    "tags": ["contract"],
    "metadata": {},
    "total_nodes": 15,
    "score": 0.94,
    "highlights": ["Payment is due within 30 days..."],
    "answer": "Payment is due within 30 days of receiving the invoice.",
    "confidence": 0.94
  }],
  "total_count": 1,
  "execution_time_ms": 1250
}
```

### Mutation Response (`/v1/query` — UPDATE / DELETE)

```json
{
  "rows_affected": 5,
  "execution_time_ms": 32
}
```

### Error Response

```json
{
  "error": {
    "code": "NOT_FOUND",
    "message": "Document not found: doc_invalid"
  }
}
```

## Resource IDs

ReasonDB uses prefixed IDs for clarity:

| Prefix | Resource | Example |
|--------|----------|---------|
| `tbl_` | Table | `tbl_7f3a9b2c` |
| `doc_` | Document | `doc_abc12345` |
| `node_` | Node | `node_xyz98765` |
| `rdb_` | API Key | `rdb_live_xxxxx` |

## Health Check

```bash
curl http://localhost:4444/health
# Returns: OK
```

## Rate Limiting

When enabled, the API returns rate limit headers:

```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 95
X-RateLimit-Reset: 1640000000
```

<Warning>
Exceeding rate limits returns `429 Too Many Requests`. Implement exponential backoff in production applications.
</Warning>
